================================================================================
                          DELIVERYHUB — ПРЕЗЕНТАЦИЯ ПРОЕКТА
                    Агрегатор для службы доставки (Fullstack)
================================================================================

СЛАЙД 1: О ПРОЕКТЕ
--------------------------------------------------------------------------------

DeliveryHub — полнофункциональный агрегатор для службы доставки.
Fullstack-приложение с аутентификацией, управлением организациями, заказами,
корзинами и адресами на картах.

Параметр            | Значение
--------------------|----------------------------------------------------------
Бэкенд              | NestJS 11, TypeScript, PostgreSQL, Prisma 6
Фронтенд            | Next.js 15 (App Router), React 19, Tailwind CSS 4
База данных         | PostgreSQL, 11 моделей, 2 enum'а
Код                 | ~47 файлов бэкенд + ~70 файлов фронтенд
Репозиторий         | https://github.com/OMI1810/DeliveryHub


СЛАЙД 2: СИСТЕМА АУТЕНТИФИКАЦИИ
--------------------------------------------------------------------------------

Реализовано:

  • Email + Password — регистрация и вход, хеширование паролей через argon2
  • JWT Access Token (1 час) + Refresh Token (7 дней, httpOnly cookie)
  • Верификация email по ссылке с одноразовым токеном
    (React Email шаблоны + SMTP через nodemailer)
  • SMS/WhatsApp OTP — 4-значный код через Infobip API (5 минут TTL)
  • Автоматический refresh токенов:
    — На клиенте: axios interceptor ловит 401, вызывает /auth/access-token
    — На сервере: middleware.ts верифицирует JWT через jose и обновляет

Роли пользователей (enum Role):

  GOD          — супер-администратор, полный доступ ко всему
  MODERATOR    — модератор, доступ к организациям и управлению
  OWNER        — владелец организации (назначается автоматически при создании)
  CLIENT       — клиент по умолчанию при регистрации
  DELIVERYMAN  — курьер (зарезервирована, логика в разработке)


СЛАЙД 3: УПРАВЛЕНИЕ ОРГАНИЗАЦИЯМИ (OWNER PANEL)
--------------------------------------------------------------------------------

Функционал:

  • Создание нескольких организаций одним пользователем (связь 1:N)
  • Панель владельца: /owner — список организаций (минималистичная сетка)
  • Страница организации: /owner/organization/:id
    — Название, email, телефон
    — Заглушка для будущих модулей (рестораны, меню, заказы)
  • Автоматическое назначение роли OWNER при создании организации
  • Валидация телефона с авто-форматированием:
    Ввод:  +79991234567 → Отображение: +7 999 123 45 67
  • Компонент PhoneInput с авто-подстановкой пробелов и валидацией
  • Валидация телефона на бэкенде через @Matches(/^\+?\d{10,15}$/)
  • Проверка доступа: только владелец / GOD / MODERATOR

Маршруты:
  /owner                    — список организаций
  /owner/create             — форма создания организации
  /owner/organization/:id   — страница организации (не модалка!)


СЛАЙД 4: СИСТЕМА ЗАКАЗОВ
--------------------------------------------------------------------------------

Модель:

  • Заказы (Order) со статусами:
    CREATED → COOKING → FROM_DELIVERYMAN → DELIVERED
  • OrderProduct — связь заказ ↔ продукт с полем quantity (явная модель)
  • payStatus — флаг оплаты (boolean)

Валидация при создании заказа:

  • Ресторан существует
  • Адрес принадлежит пользователю
  • Продукты принадлежат ресторану
  • Транзакционное создание — атомарность данных

Эндпоинты:
  GET  /api/orders         — мои заказы
  GET  /api/orders/:id     — заказ по ID (проверка владения)
  POST /api/orders         — создать заказ


СЛАЙД 5: КОРЗИНА (CART)
--------------------------------------------------------------------------------

Функционал:

  • Мульти-ресторанная корзина на localStorage
  • Группировка товаров по restaurantId
  • Управление: +/- quantity, удаление товара, полная очистка
  • CartIcon с бейджем количества (subscribe/notify паттерн)
  • CheckoutForm — модалка оформления заказа:
    — Выбор адреса из списка
    — Создание Order через API
  • Хук useCart() — полное управление корзиной:
    carts, totalItems, addToCart, updateQuantity,
    removeFromCart, clearCart, clearAll, getCartTotal

Компоненты:
  CartIcon           — иконка корзины с бейджем
  AddToCartButton    — кнопка "В корзину" с анимацией подтверждения
  RestaurantCart     — карточка корзины конкретного ресторана
  CheckoutForm       — модалка оформления заказа


СЛАЙД 6: АДРЕСА С КАРТАМИ
--------------------------------------------------------------------------------

Реализовано:

  • Leaflet карта (react-leaflet, OpenStreetMap) с кастомным маркером
  • ArcGIS геокодер — автоподсказки адресов (suggest API)
  • Точное определение адреса по magicKey (ArcGIS find)
  • Обратное геокодирование (координаты → адрес)
  • Клик по карте → обратное геокодирование → заполнение полей

Поля адреса:
  • Адрес (строка)
  • Квартира, этаж, подъезд, домофон
  • Комментарий
  • Координаты (cordinatX, cordinatY)

CRUD с проверкой владения — пользователь видит только свои адреса

Эндпоинты:
  GET    /api/users/addresses          — все адреса пользователя
  GET    /api/users/addresses/:id      — конкретный адрес
  POST   /api/users/addresses          — создать адрес (автогеокодирование)
  PATCH  /api/users/addresses/:id      — обновить адрес
  DELETE /api/users/addresses/:id      — удалить адрес


СЛАЙД 7: ROLE-BASED ACCESS CONTROL
--------------------------------------------------------------------------------

Уровень        | Механизм
---------------|---------------------------------------------------------------
Бэкенд         | @Auth(Role.GOD), RolesGuard — проверка ролей из JWT payload
Фронтенд       | middleware.ts + jose JWT verify на сервере
Редиректы      | Неавторизованные → /404 (admin) или /auth/login (dashboard)

Защита маршрутов (middleware.ts):

  /auth/*        → protectLoginPages: если авторизован → редирект на /dashboard/profile
  /admin/*       → protectAdminPages: требует GOD роль, иначе → 404
  /manager/*     → protectAdminPages: требует OWNER или MODERATOR, иначе → 404
  /dashboard/*   → protectDashboardPages: требует любой валидный токен
  /owner/*       → protectDashboardPages: требует любой валидный токен

Декораторы на бэкенде:
  @Auth()              — требует JWT (JwtAuthGuard)
  @Auth(Role.GOD)      — требует JWT + роль GOD
  @CurrentUser('id')   — извлекает поле из текущего пользователя
  @Roles(roles)        — устанавливает метаданные для RolesGuard

СЛАЙД 10: БАЗА ДАННЫХ (Prisma Schema)
--------------------------------------------------------------------------------

Модели (11):

  User              — основной пользователь (email, phone, password, ФИО)
  UserRole          — роли пользователя (уникальная пара userId + role)
  Shift             — смены курьеров (createAt, stopAt)
  Address           — адреса с координатами
  AddressUser       — связь User ↔ Address (уникальная пара)
  AddressRestaraunt — связь Restaurant ↔ Address
  Organization      — организация-владелец ресторанов
  Restaurant        — ресторан (название, кухня, время работы)
  Product           — продукт ресторана (название, калории, время готовки)
  Order             — заказ (статус, оплата, клиент, ресторан, адрес)
  OrderProduct      — связь Order ↔ Product с полем quantity

Enum'ы (2):

  Role     — GOD, MODERATOR, CLIENT, OWNER, DELIVERYMAN
  Status   — CREATED, COOKING, FROM_DELIVERYMAN, DELIVERED

Ключевые связи:
  User 1--* UserRole (CASCADE)
  User 1--* AddressUser (CASCADE) *--1 Address
  User 1--* Organization (1:N — владелец может иметь много организаций)
  User 1--* Order (клиент)
  Organization 1--* Restaurant
  Restaurant 1--* Product
  Restaurant 1--* Order
  Order *--* Product через OrderProduct (с quantity)

Миграции (4):
  20260404105221_move_roles_to_table               — роли в отдельной таблице
  20260404171907_remove_unique_owner_id_on_organizations — 1:N User↔Organization
  20260404170205_add_order_product                  — явная модель OrderProduct
  20260404185021_update_order_product                — индексы

СЛАЙД 12: АРХИТЕКТУРНЫЕ ПАТТЕРНЫ
--------------------------------------------------------------------------------

  Module-based (NestJS)    — 8 модулей, каждый отвечает за свою область
  Guard/Decorator          — @Auth(), RolesGuard, JwtAuthGuard
  Strategy (Passport)      — JwtStrategy для JWT аутентификации
  Repository (Prisma)      — PrismaService как singleton, типобезопасные запросы
  Service Layer            — отдельные сервисы для каждой доменной области
  DTO + ValidationPipe     — все входные данные валидируются через class-validator
  Transaction              — создание адреса и заказа — в $transaction
  Feature-Sliced (частично)— app/, components/, services/, hooks/, types/
  App Router (Next.js)     — app/ директория, layout.tsx, page.tsx, metadata
  Middleware Pattern       — middleware.ts для route protection
  Interceptor Pattern      — axios: auto-refresh токенов по 401
  Pub/Sub                  — CartIcon: subscribeToCartChanges() / notifyCartChange()
  SSR/CSR Mix              — server-variation.tsx для серверного рендера,
                             dynamic() для SSR-safe карт


СЛАЙД 13: КЛЮЧЕВЫЕ РЕШЕНИЯ И ИНВАРИАНТЫ
--------------------------------------------------------------------------------

  • Refresh Token в httpOnly cookie — защита от XSS
  • Access Token в Authorization header — стандартный подход
  • ignoreExpiration: true в JwtStrategy — проверка на уровне middleware
  • PhoneInput: форматирование на клиенте, cleanPhone() перед отправкой
  • Корзина на localStorage — не требует авторизации
  • OrderProduct — явная модель (вместо implicit m2m) для поля quantity
  • Организация: 1:N с User (один пользователь — много организаций)
  • Маршрутизация вместо модалок — /owner/organization/:id как отдельная страница
  • Prisma: @@unique для составных ключей, @@index для производительности
